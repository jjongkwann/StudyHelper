# 인덱스 아키텍처

## B+Tree 인덱스

B+Tree는 관계형 데이터베이스에서 가장 널리 사용되는 인덱스 구조이다.

- 리프 노드에만 실제 데이터 포인터 저장
- 리프 노드끼리 Linked List로 연결 → 범위 검색에 유리
- 검색 복잡도: O(log N), 풀 스캔 O(N) 대비 효율적
- 디스크 I/O 최소화: 높은 팬아웃(fanout)으로 트리 높이 최소화

## 클러스터형 vs 비클러스터형 인덱스

- **클러스터형**: 데이터 자체가 인덱스 순서로 물리적 정렬. 테이블당 1개만 가능
- **비클러스터형**: 별도의 인덱스 구조에 데이터 포인터 저장. 여러 개 가능

## 인덱스 설계 트레이드오프

- 읽기 성능 향상 vs 쓰기 성능 저하 (INSERT/UPDATE 시 인덱스도 갱신)
- 과도한 인덱스 → 페이지 분할(Page Split) → 성능 저하
- 복합 인덱스: 선두 컬럼의 선택도(Selectivity)가 높아야 효과적
- 커버링 인덱스: 쿼리에 필요한 모든 컬럼이 인덱스에 포함 → 테이블 접근 불필요
